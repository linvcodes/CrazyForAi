Layout requirements

A vertical card centered on the page that feels like a "digital business card."

The card container must be ~90% of viewport height (min-height: 90vh) with margins around it so it never touches edges.

Inside the card: photo at top, then name/title/tagline, then icon row, then big Portfolio button.

CRITICAL — Responsive rules (no exceptions):
- Mobile-first: design for small screens first, then scale up
- Zero hardcoded px values for widths, heights, font sizes, or spacing
- All font sizes: rem or em only
- All spacing (padding, margin, gap): rem or em only
- Layout widths: %, vw, max-width with %, or clamp()
- Use flexbox or grid — never fixed pixel positioning
- The card must look finished and complete on first load on any screen size

How to Turn This Into a Native App

Option 1 — Wrap It With Capacitor

Web → Native Shell

What it is

Use Capacitor to package the existing website as a native Android (APK) and iOS app.

Pros
  • Fastest path to native
  • Reuses existing web code
  • Single codebase
  • Access to device APIs via plugins
  • Can publish to App Store / Play Store
  • Lower cost to maintain
Cons
  • Still web-rendered (WebView-based)
  • Slight performance limitations for heavy 3D/games
  • Native feel depends on web UI quality
Best for: MVPs, SaaS apps, dashboards, content apps, internal tools.
Option 2 — Rebuild in Native

Flutter / React Native / Swift / Kotlin

What it is

Rebuild the app using a true native or cross-platform mobile framework.

Pros
  • Maximum performance
  • True native UI components
  • Full hardware access
  • Better for complex animations / heavy graphics
  • More scalable long-term
Cons
  • Higher development time
  • Higher cost
  • Separate mobile architecture
  • Slower iteration vs web/PWA
Best for: High-performance apps, games, hardware-intensive features, long-term mobile-first products.

Capacitor is ideal when you already have a working web product and want rapid store deployment. Rebuilding native is best when performance, hardware integration, or long-term mobile scalability is critical.

Cut Your AI Bill by 10× — Token Cost Strategy

ZhipuAI — Claude-Compatible API

ZhipuAI (智谱AI) offers GLM-4 models with Claude-compatible API format at a fraction of the cost. Useful for China-accessible deployments or simply cutting prototype costs to near zero.

GLM-4-Flash: free tier available OpenAI-compatible endpoints bigmodel.cn
Real cost example

1,000 user messages/day × 500 tokens avg = 500K tokens/day.
Haiku: ~$0.40/day · Sonnet: ~$1.50/day · GPT-4o: ~$5/day

5 Ways to Save on Tokens
  1. System prompt caching — Anthropic charges 10% to cache a prompt and 0% to reuse it. Pin your system prompt once per session.
  2. Trim context aggressively — every token in the conversation history costs money. Summarise old turns instead of appending forever.
  3. Use the smallest model that works — Haiku is 15× cheaper than Sonnet. Route simple tasks there first, escalate only on failure.
  4. Batch where latency doesn't matter — Anthropic Batch API is 50% cheaper for async jobs (reports, email drafts, nightly syncs).
  5. Stream + abort early — if the first 100 tokens already answer the question, cancel the stream. You only pay for what's generated.

Your MVP as a Playable Ad — Test Before You Build

A playable ad is a self-contained HTML/CSS/JS file that runs inside any ad network — Meta, Google, TikTok, IronSource. It's also the fastest way to validate a product idea: ship a 30-second interactive demo to 10,000 users before writing a single line of your real backend.

What you can measure
  • Tap / interaction rate — do users engage with the core mechanic?
  • Drop-off point — where do they stop?
  • CTA conversion — how many click "Download" / "Sign Up"?
  • Variant A/B — swap one variable per ad creative, measure lift
  • Audience targeting signal — Meta will find your high-intent users
How to build one in an afternoon
  1. Build your core mechanic in vanilla HTML/CSS/JS (Claude does this in minutes)
  2. Add a postMessage("click") call on your CTA button — ad networks intercept this to trigger store redirect
  3. Inline all assets (base64 images, embedded fonts) — the ad must be a single self-contained file
  4. Add 5 lines of analytics: timestamp, interaction count, drop-off event, CTA fired
  5. Upload to Meta Ads Manager → Playable format → run with $20 test budget
What a 40% interaction rate tells you

In mobile, 40%+ interaction rate on a playable ad = product-market signal. You've validated the core mechanic resonates before spending a month building it. Under 15% = rethink the mechanic, not the marketing.

>40% interact = build it 15–40% = iterate mechanic <15% = wrong idea

What an MVP Actually Needs

The best MVP isn't the smallest thing you can build — it's the smallest thing that still proves your core assumption. One critical user action (the "aha moment"). Everything else is noise.

What's in
  • Core loop — user does something, sees a result. That's the whole product for now.
  • Barebones UI — clickable and understandable. Not polished. Ship ugly.
  • Simple data — localStorage, Firebase, or hardcoded. No custom backend yet.
  • Analytics hook — PostHog or even just a Google Sheet via a fetch call. You must learn what users actually do.
  • One feedback path — Tally form, WhatsApp group, or "reply to this email." Talk to your first 50 users directly.
What's out
  • Auth (use localStorage until users ask for it)
  • Scalability & edge cases
  • Settings, preferences, profiles
  • Mobile app (PWA is enough)
  • Custom design system
  • A pitch deck before a working product

For games or playable ads, the MVP can be even tighter — a single 10–20 second loop that proves retention or click-through is enough to decide whether it's worth scaling.

Best stack to start
  1. Vanilla HTML / CSS / JS — fastest path to something clickable. No setup, no architecture decisions, no framework opinions.
  2. Vite — instant dev server, hot reload, zero config. Add it when vanilla feels slow.
  3. No backend — localStorage or mock data until it becomes painful.

Only upgrade when you feel the pain: UI getting complex → add React. Need shared state → hooks. Need data persistence → Supabase. Speed of iteration beats everything at this stage.
